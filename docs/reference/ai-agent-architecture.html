A
🧠
最小化LLM约束
只给LLM定义"硬边界"，其余全部开放。不用大段系统提示控制行为，而是让LLM自主推理最优路径。
  • 系统提示仅包含：角色定义 + 工具清单 + 硬边界规则
  • 不预设思维方式，让LLM自主选择 ReAct / ToT / CoT
  • 目标描述清晰，过程不干预
  • 允许LLM拒绝不合理的子任务并重新规划
B
🎯
目标结果导向
评估一切行为的唯一标准是"是否更接近目标"。终止条件由目标决定，而非步骤数或时间限制。
  • Goal DSL：结构化描述目标 + 成功指标 + 截止条件
  • 每轮迭代后重新评估距目标的距离
  • 支持目标分层：主目标 → 子目标 → 即时任务
  • 失败也是有效终止，触发复盘流程
C
⚙️
工具优先执行
LLM负责决策，工具负责执行。能用工具解决的绝不靠LLM幻觉，工具结果作为事实输入回到上下文。
  • 工具注册表：动态发现+注册新工具
  • 工具选择器：根据任务自动匹配最优工具
  • 并行工具调用：独立子任务同时执行
  • 工具失败降级：自动寻找替代工具

04 · Memory Architecture

四层记忆体系

⚡ 工作记忆
当前任务上下文窗口。包含本轮对话、工具调用链、中间变量。随任务结束清空。
Context Window · Token管理 · 压缩摘要
📼 情节记忆
历史任务的完整经历快照。"上次做X时，用了Y方法，结果Z"。语义向量检索。
Vector DB · Pinecone/Weaviate · RAG召回
🕸️ 语义记忆
结构化知识图谱，存储实体、关系、概念。跨任务的领域知识积累与连接。
Knowledge Graph · Neo4j · 实体抽取
🎯 程序记忆
已验证的技能库、成功模板、工具调用序列。"解决X类问题的标准SOP"。直接复用。
Skill Library · Template Store · 版本管理
// 记忆读写策略 class MemorySystem { async recall(query: string, k = 5) { // 1. 程序记忆优先（精确匹配已知技能） const skill = await this.skillLibrary.match(query); if (skill.confidence > 0.9) return { type: 'skill', data: skill }; // 2. 情节记忆次之（相似历史经验） const episodes = await this.vectorDB.search(query, k); // 3. 知识图谱补充（背景领域知识） const entities = await this.knowledgeGraph.query(query); return { episodes, entities }; } async consolidate(episode: Episode) { // 任务结束后异步提炼知识写入长期记忆 const insights = await llm.extract(episode, "what did we learn?"); await this.vectorDB.upsert(episode); await this.knowledgeGraph.merge(insights.entities); if (episode.success) await this.skillLibrary.register(episode.workflow); } }

1
执行信号采集
每次任务执行结束后，自动记录：完成状态、使用的工具链、步骤数、耗时、失败节点、用户反馈信号。构成原始经验数据。
2
失败根因复盘
对失败/低效任务做自动归因分析：是工具选择错误？规划路径不优？信息获取不足？上下文理解偏差？生成结构化故障报告。
3
知识蒸馏入库
LLM对成功经验做知识提炼，生成可复用的：技能SOP、工具调用模板、领域知识摘要。写入程序记忆和语义记忆。
4
工具能力扩展
当遇到反复失败的任务类型时，智能体可自动编写新工具代码（在沙箱中测试通过后），注册到工具库，下次遇到同类任务直接使用。
5
策略参数更新
调整任务规划偏好、工具优先级权重、并行度阈值、失败重试策略等配置参数，形成个性化的"最优实践"配置文件。
6
A/B验证回测
新策略不直接全量替换，而是在下一批任务中进行对比测试。只有显著优于旧策略时才正式启用，防止进化方向跑偏。
06 · Hard Boundaries

硬边界设计原则

哲学： 少即是多。硬边界不是枷锁，是智能体可信赖的根基。边界越少越精准，LLM的自主空间越大。只限制"不可逆的破坏性行为"，其余全部放开让智能体自决。
充分自由：LLM可以自主决定
  • 选择什么工具、调用几次、以何种顺序
  • 规划路径、分解方式、中间步骤
  • 是否需要更多信息、是否重新理解目标
  • 选择什么编程语言、框架、技术方案
  • 何时并行何时串行执行
  • 如何格式化输出、用何种表达方式
  • 是否认为当前目标可以优化，并提出建议
  • 创建临时子任务或中间产物
硬性边界：绝对不可越过
  • 不可删除用户明确标记的受保护数据
  • 不可在未经确认的情况下做不可逆的外部操作（发送邮件、提交代码到生产、删除数据库）
  • 不可主动向外部系统发送用户隐私数据
  • 单次任务的工具调用次数不超过硬上限（防止死循环）
  • 不可修改自身的硬边界规则
  • 财务类操作超过阈值必须人工审批
  • 不可绕过审计日志（所有动作必须可追溯）
// 硬边界实现：不可绕过的守卫层（独立于LLM） class HardBoundaryGuard { private readonly rules: ImmutableRuleSet; // 不可运行时修改 intercept(action: ToolCall): Decision { // 1. 不可逆操作检查 if (this.isIrreversible(action) && !action.hasHumanApproval) return { allow: false, reason: 'NEEDS_HUMAN_APPROVAL' }; // 2. 资源限制检查 if (this.callCount > this.rules.maxToolCalls) return { allow: false, reason: 'CALL_LIMIT_EXCEEDED' }; // 3. 数据安全检查 if (this.containsPII(action) && action.target === 'external') return { allow: false, reason: 'PII_LEAK_PREVENTION' }; // 4. 写入不可篡改审计日志 this.auditLog.append({ action, timestamp, agentId, sessionId }); return { allow: true }; } }

08 · Design Principles

十条设计原则

1
🎯
目标是唯一仲裁者
一切行为、工具选择、路径规划，评估标准只有一个：是否更接近目标。不为过程打分，只为结果打分。
2
🔓
最大化LLM自由度
不用规则约束LLM的思维方式。只定义"不能做什么"，不规定"怎么做"。过多的Prompt约束会扼杀智能涌现。
3
🔧
工具是能力边界
智能体能做什么 = 它拥有的工具集。不断扩展工具库（包括自动生成新工具）是提升能力的核心路径。
4
⚡
失败是最好的数据
每次失败都是高质量的训练信号。完整记录失败路径，自动归因，将"怎么不能做"转化为未来的避坑知识。
5
🛡️
硬边界必须独立
安全守卫不依赖LLM实现，而是独立的代码层。LLM绝不能说服自己绕过安全规则，因为规则根本不经过LLM。
6
🧩
记忆分层管理
不同类型的知识存在不同层次的记忆中。上下文窗口宝贵，只放最相关的信息；长期记忆按需按序召回。
7
🔄
进化要有验证
新学到的策略不直接覆盖旧策略。必须在受控条件下A/B测试，证明更优后才正式采用，防止"越学越差"。
8
👁️
全程可观测
每一个决策节点、每一次工具调用、每一轮迭代都留有追踪记录。黑盒智能体无法信任，也无法优化。
9
🤝
不可逆操作人工确认
对于发送邮件、删除数据、提交生产代码等不可逆的高影响操作，即使智能体高度自信，也需要人工一次性确认。
10
🌐
通用性源于抽象
不针对特定领域定制提示词，而是通过通用的目标-工具-记忆抽象，让同一架构适配任何垂直场景，只需更换工具集。
