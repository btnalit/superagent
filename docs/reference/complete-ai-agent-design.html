01 · Philosophy

设计哲学：简单是结果，不是捷径

大多数系统之所以复杂，是因为没想清楚职责边界，所以用复杂度弥补设计的模糊。
当每个组件只回答一个问题时，系统自然就小了。
"该改什么" 和 "能不能改" 天然不应该由同一个系统回答——
前者需要创造力和自由度，后者需要保守和确定性。
①
目标是唯一仲裁者
评估一切行为只有一个标准：是否更接近目标。不为过程打分，只为结果打分。终止条件由目标决定，不由步骤数决定。
②
职责边界只有一条线
两个系统之间有且只有一条信任边界。没有侧门，没有快捷通道。信任边界的价值不在于它有多复杂，在于它是不是唯一路径。
③
最大化 LLM 自由度
只定义"不能做什么"，不规定"怎么做"。过多的 Prompt 约束会扼杀智能涌现。系统提示只包含：角色定义 + 工具清单 + 硬边界。
④
Skills 是动态资产，工具是静态基础
工具几乎不变；Skills 持续生长。两个拥有相同工具集的智能体，Skills 库决定了能力的天花板。要对接新系统，建 Skill，而不是改架构。
⑤
进化必须可验证
新策略不直接覆盖旧策略。必须在受控条件下 A/B 验证，证明更优后才正式启用。失败是最好的训练信号，但进化不能跑偏。
02 · Architecture

六层架构：每层只回答一个问题

LAYER 01
Goal Layer
意图解析 任务分解 成功指标定义 子目标树生成 优先级排序 → 回答：我要到哪里去？
LAYER 02
Plan Layer
ReAct 推理循环 动态重规划 依赖图管理 并行任务调度 死锁检测 → 回答：怎么走最优？
LAYER 03
LLM Core
上下文管理 思维链推理 工具调用决策 结果评估 自我反思 最小系统提示 → 回答：下一步做什么？
LAYER 04
Skill + Tool Layer
Skill 调用与编排 Code Runner HTTP Caller Search+Fetch File System Memory R/W Sub-Agent → 回答：怎么执行？
LAYER 05
Evolution Layer
self-evolution-governor 经验提炼 失败复盘 Skill 生成进化 提案生成 → 回答：该改什么？
LAYER 06
Hard Boundary
ops-gate（独立于 LLM） precheck 条件验证 不可逆操作拦截 不可篡改审计日志 沙箱隔离 人工确认钩子 → 回答：能不能改？

// 核心循环伪代码 — 简洁是刻意的 async function agentLoop(goal: Goal) { const state = initState(goal); while (!isGoalMet(state)) { // Think — LLM 自由推理，不加约束 const plan = await llm.reason(state, tools.registry, memory.recall(state)); // Act — 并行执行，全部经过 ops-gate const results = await Promise.all( plan.actions.map(a => opsGate.intercept(a).then(execute)) ); // Observe + Update state.update(results); memory.write(state.episode); // 防止死循环的唯一硬上限 if (state.iterations > HARD_LIMIT) break; } // 任务结束 → 触发异步进化（不阻塞结果） evolutionGovernor.consolidate(state.episode); }
04 · The Two-System Design

双系统设计：各管各的事

核心洞察： "该改什么"和"能不能改"天然不应该由同一个系统回答。把两个问题放在一起，要么创造力被保守压死，要么安全被自由绕过。分开，两个系统都能做到极致。

// ops-gate：独立于 LLM 的硬边界守卫 // LLM 永远无法"说服"它绕过规则，因为规则根本不经过 LLM class OpsGate { private readonly rules: ImmutableRuleSet; // 运行时不可修改 intercept(action: ToolCall): Decision { // ① precheck：条件齐不齐 if (!this.precheck(action)) return REJECT('preconditions_not_met'); // ② 不可逆操作：必须有人工批准凭据 if (this.isIrreversible(action) && !action.humanApproval) return HOLD('awaiting_human_approval'); // ③ 迭代硬上限：防止死循环 if (this.callCount >= this.rules.MAX_CALLS) return REJECT('hard_limit_exceeded'); // ④ 必须写入不可篡改审计日志（evidence） this.auditLog.append({ action, ts: Date.now(), agentId, sessionId }); return ALLOW; } }
05 · Skills — The Main Thread

Skills 是主线：动态生长的智能资产

核心认知： Skills ≠ 工具调用。Skills 是封装了"解决某类问题全部知识"的完整工作流单元——包括何时用哪个工具、遇到什么情况怎么处理、输出什么格式、如何判断成功。要对接新系统？建一个 Skill。
1
发现
遇到没有现成 Skill 的任务，触发 Skill Forge
2
生成
LLM 分析任务类型，自动拟定工具链和执行策略
3
验证
沙箱执行 3–5 次，采集成功率、错误、耗时
4
入库
成功率 > 80% 进入技能库，可被其他 Skill 调用
5
进化
每次执行采集信号，A/B 竞争，永不停止迭代

06 · Universal Tools

10 个工具：覆盖 95% 场景

设计原则： 工具粒度要"恰到好处"。太细（每个 API 一个工具）→ 注册表爆炸，LLM 选择困难；太粗 → 失去精确控制。工具要精，但每个要深。每增加一个工具，LLM 选择复杂度就上升。
工具 能力 · 用途 · 为什么必须有 优先级
💻
Code Runner
CORE
沙箱执行任意代码（Python/JS/Bash），返回 stdout / stderr / 文件产物。理论上可替代其他所有工具，但用它模拟其他工具太重，所以只做计算和处理逻辑。
→ 有了它就有无限计算能力，是 LLM 从"说话者"变为"行动者"的核心
★★★★★
🌐
HTTP Caller
CORE
带认证管理的 HTTP/REST/GraphQL，支持 OAuth / API Key / Bearer 自动注入，响应解析全自动。
→ 有了它就连通万物，是突破单机局限的关键通道
★★★★★
🔍
Search + Fetch
CORE
网络搜索 + 网页完整内容提取，支持动态页面（Playwright），自动 Markdown 化。
→ 突破 LLM 知识截止日期，实时信息获取不可或缺
★★★★★
📁
File System
CORE
文本/JSON/CSV/PDF/Office 读写，大文件分块处理，支持云存储挂载（S3/GCS）。
→ 任务中间产物和最终输出的持久化载体，智能体的"硬盘"
★★★★☆
🧠
Memory R/W
CORE
向量检索 + KV 存取 + 知识图谱，统一封装四层记忆（工作/情节/语义/程序），写入自动生成 embedding。
→ 跨会话学习和知识复用的根基，没有它智能体每次都从零开始
★★★★★
🤖
Sub-Agent
POWER
启动专用子智能体处理子任务，同步等待或异步回调，子 Agent 共享父级工具权限（可限制）。
→ 复杂任务分治，是系统扩展性的核心，让智能体可以无限延伸
★★★★☆
🗄️
Database
POWER
直连 SQL/NoSQL（PostgreSQL/MySQL/MongoDB/Redis），自然语言 → SQL，结果自动序列化。
→ 处理结构化数据比绕道 HTTP API 快 10 倍，企业场景刚需
★★★★☆
📬
Notify Hub
POWER
统一发送接口（Email/Slack/钉钉/Webhook/SMS），也接收回调，是人工审批流的实现载体。
→ 不可逆操作的人工确认钩子，ops-gate 的通知通道
★★★☆☆
👁
Vision + OCR
POWER
图像/截图理解 + 文字提取，扫描件 OCR，图表解读，UI 状态识别，让 Agent 能"看"。
→ 大量现实数据以图片存在，没有视觉能力会遇到大量瓶颈
★★★★☆
🔧
Skill Forge
META
元工具：自动生成、测试、注册新 Skill。当反复解决同类问题时，自动提炼成功工作流并写入技能库。
→ 自进化的关键，让 Skills 从 0 生长，智能体越用越强
★★★★★
07 · The Formula

能力公式：简洁的方程，深刻的含义

Agent Capability = Tools（地基，几乎不变）
             ×  Skills（资产，持续生长）
             ×  Evolution Loop（斜率，决定上限速度）
             ÷  Constraints（越少越好，只保留硬边界）
工具是下限 · Skills 是上限 · 进化闭环决定斜率 · 约束是分母
1
每个组件只回答一个问题
系统复杂度的根源是"职责不清"。当每个模块只负责一个问题时，复杂度自然收敛。不要用复杂度弥补设计的模糊。
2
信任边界有且只有一条线
governor → 用户批准 → ops-gate。没有侧门，没有快捷通道。 边界的价值不在于它复杂，在于它是唯一路径。
3
硬边界必须独立于 LLM
ops-gate 是代码层，不是 Prompt 层。LLM 永远无法"说服"规则，因为规则根本不经过 LLM。 这才是真正的安全。
4
Proposal 是纯数据，不是可执行代码
governor 的输出可以被暂停、被审计、被拒绝、被人介入，不产生任何副作用。这才是"人在回路"的正确实现。
5
进化在运行中发生，但不在进化中崩溃
governor 只动 state/evolution/Skills，不碰正式系统。系统可以在运行中学习，但不会在学习中崩溃。
6
失败是最好的数据，但进化要有验证
新策略不直接覆盖旧策略，必须 A/B 验证。防止越学越差是自进化系统的最大陷阱。
7
全程可观测，黑盒无法信任
每个决策节点、每次工具调用、每轮迭代都留有追踪。可观测性不是可选项，是 ops-gate 审计能力的前提。
